# Lab 9 — Stateful Services & DB Reliability

## Task 1 — Migrations and Backup/Restore

### Alembic Setup

I initialized Alembic and configured it for the QuickTicket database.

The following steps were completed:

* Initialized Alembic using `alembic init migrations`
* Created a baseline revision for the existing database schema
* Created a migration that adds an `email` column to the `events` table

### Migration Under Load

While the migration was running, the application continued serving requests generated by the mixed load deployment.

Migration applied:

```sql
ALTER TABLE events ADD COLUMN email VARCHAR(255);
```

The migration completed successfully without interrupting the application. Since the new column was nullable, PostgreSQL was able to apply the schema change safely under load.

No additional application errors were observed during the migration.

### Backup and Restore

A database backup was created using:

```bash
pg_dump -Fc > /tmp/quickticket.dump
```

To simulate data loss, the `orders` table was removed:

```sql
DROP TABLE orders CASCADE;
```

The database was then restored using:

```bash
pg_restore --clean --if-exists
```

After the restore completed, the database schema and application functionality were successfully recovered.

### RPO Observation

The recovery process demonstrated the following:

* Before the simulated failure, the `orders` table was present.
* After executing `DROP TABLE`, the table was no longer available.
* After restoring the backup, the table was successfully recovered.

The Recovery Point Objective (RPO) depends on the time elapsed since the latest backup. Any data created after the backup would be lost in the event of a disaster.

# Task 2 — Disaster Recovery Under Load

## Experiment

To simulate a database failure, the PostgreSQL pod was deleted:

```bash
kubectl delete pod -l app=postgres --grace-period=0 --force
```

Kubernetes automatically created a replacement pod.

Because the PostgreSQL deployment was using ephemeral storage, the recreated pod started with an empty database.

The database was restored from the previously created backup, after which the Events service was restarted to re-establish database connections.

```bash
kubectl rollout restart deployment/events
```

After the restart, the application became operational again.

### RTO and RPO

* **Recovery Time Objective (RTO):** approximately 1.5 to 2 minutes, including pod recreation, database restoration, and application restart.
* **Recovery Point Objective (RPO):** determined by the age of the most recent backup. In this setup, the potential data loss was limited to changes made after the latest backup.

### Conclusion

This experiment demonstrated that PostgreSQL running without a PersistentVolumeClaim is vulnerable to complete data loss after pod recreation. While backups allow recovery, they increase recovery time and may result in data loss between backup intervals.

# Bonus Task — Persistent Storage and Automated Backup

To improve reliability, a PersistentVolumeClaim (PVC) was added to the PostgreSQL deployment.

Allocated storage:

```yaml
storage: 1Gi
```

In addition, automated database backups were configured using a Kubernetes CronJob with the following policy:

* Backup executed every five minutes.
* Retention policy keeps the five most recent backup files.

### Disaster Recovery After Adding Persistent Storage

After introducing persistent storage, recreating the PostgreSQL pod no longer resulted in data loss.

The database remained available after the pod restarted, eliminating the need to restore from backup during normal pod failures.

As a result, recovery time was significantly reduced and was limited primarily to the PostgreSQL startup time.

# Final Thoughts

This lab provided practical experience with database reliability and disaster recovery in Kubernetes.

The key concepts covered included:

* Alembic database migrations
* Performing schema migrations under application load
* Creating and restoring PostgreSQL backups using `pg_dump` and `pg_restore`
* Measuring Recovery Time Objective (RTO) and Recovery Point Objective (RPO)
* Using PersistentVolumeClaims to preserve database data
* Automating backups with Kubernetes CronJobs

The most important lesson from this lab is that stateful applications should always use persistent storage. Running a database on ephemeral container storage creates a significant risk of data loss after pod recreation. Combining persistent storage with regular automated backups provides a much more reliable and resilient deployment.
